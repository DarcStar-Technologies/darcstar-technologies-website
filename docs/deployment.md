# Deployment ‚Äî Cloudflare

SvelteKit via `@sveltejs/adapter-cloudflare`, deployed to **Cloudflare Pages**. Build output goes to `.svelte-kit/cloudflare`.

- Bindings/env are typed through `Env` (generated by `wrangler types`) and surfaced on `App.Platform` in `src/app.d.ts`.
- `compatibility_flags: ["nodejs_als"]` is set in `wrangler.jsonc`.
- Run `pnpm gen` after changing `wrangler.jsonc` bindings to regenerate `worker-configuration.d.ts`. `wrangler types` also reads secret **names** from `.env`, so a new secret needs a line in `.env` (local) + `.env.example` (committed template) before its `platform.env.*` access is typed.
- The `wrangler-types` pre-commit hook enforces this, but is **scoped to commits that touch `wrangler.jsonc`** ‚Äî it no longer fires on every commit. The committed `worker-configuration.d.ts` is the **build-independent** snapshot (no `mainModule` block, which only appears after `vite build`), so `pnpm gen` on a clean checkout reproduces it byte-for-byte and unrelated commits don't trip the hook (issue #67).
- `pnpm preview` serves the built worker through Wrangler ‚Äî a real Workers runtime, not `vite preview`. See [commands.md](commands.md).

## Secrets

Non-public values are set with `wrangler secret put <NAME>` (not in `wrangler.jsonc`), mirrored in local `.env`:

- `DATABASE_URL`, `DATABASE_AUTH_TOKEN` ‚Äî Turso.
- `BETTER_AUTH_SECRET`, `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET` ‚Äî auth.
- `RESEND_API_KEY` ‚Äî contact lead notifications (issue #52). **Also requires** verifying `darcstar.tech` as a sending domain in Resend (add the DKIM/SPF DNS records to Cloudflare). Until both are done, submissions still persist ‚Äî the email send skips (no key) or logs its failure; it never fails the submission. See [contact.md](contact.md).
